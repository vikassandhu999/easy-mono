# Training Context Refactor Design

## Goal

Refactor the training context around a simple weekly-recurring coaching model that fits the current product market, while making workout logging, history, snapshots, substitutions, and API boundaries solid enough for serious coaching workflows.

This is a clean-break refactor. The application is not in production, so implementation should directly change schemas, migrations, OpenAPI schemas, controllers, contexts, tests, and factories without preserving old endpoints or old response shapes.

## Product Decision

The canonical plan model is weekly recurrence.

Indian coaches commonly give weekly schedules and adapt them manually. That covers the primary product use case better than a full periodized week/day engine. Periodization is handled by assigning a new weekly plan for the next phase, or by editing the current client plan when changes are minor.

The product promise is:

```text
Easy is a weekly coaching-plan system with excellent client-specific customization and logging.
```

Do not build `plan_weeks`, `plan_days`, or a full calendar/periodization engine now.

## Accepted Model

```text
training_plan_template
  -> copied client_training_plan
    -> weekly training_plan_items
    -> workouts
      -> workout_elements
        -> planned_sets

active client_training_plan
  -> materialized scheduled_workouts for current/past days
    -> workout_sessions
      -> performed_exercises
        -> performed_sets
```

Future schedule views are virtual. They are calculated from the client's current active weekly plan.

Current and historical workout occurrences are materialized. They preserve the plan item and workout prescription as the client saw it on that day.

## Explicit Non-Goals

- Do not add `plan_weeks`.
- Do not add `plan_days`.
- Do not add a recurrence-rule engine.
- Do not build TrainingPeaks-style periodized blocks now.
- Do not build live template synchronization.
- Do not keep old API request or response shapes for compatibility.
- Do not update `api_contract.yml`; use generated OpenAPI as the API contract.
- Do not expose `business_id` in any response.
- Do not let clients set trusted identifiers from request bodies.

## Current Problems To Fix

- Templates are inferred with `client_id == nil`.
- Assigned/client plans require dates too early, which blocks preparing plans before a client knows when they will start.
- Duplicate plan names are generated automatically with `"(Copy)"`.
- `training_plan_items` only model `primary` and `alternative`.
- `rest_days` duplicates information that can be represented by weekday plan items and makes conflicts harder.
- Future schedule and historical schedule are not clearly separated.
- Workout sessions have a `planned_snapshot`, but session creation and performed-set logging do not fully preserve planned-vs-actual history.
- Planned sets are embedded, which makes stable set identity and planned-vs-actual comparison harder.
- Performed logging is set-centric and does not clearly represent exercise substitution, skipped sets, added sets, changed order, pain, or coach review.
- OpenAPI documents training plan endpoints, but training plan request validation should be enforced at the boundary with `OpenApiSpex.Plug.CastAndValidate`.

## Target Concepts

### Training Plan Template

A reusable weekly plan owned by a coach/business.

Rules:

- Has no client.
- Has no start date.
- Is not directly visible to clients.
- Can be duplicated only with a user-provided name.
- Can be copied into a client plan draft.

### Client Training Plan

A client-specific copy of a template or a coach-created client plan.

Rules:

- Has a client.
- Can be `draft` without a start date.
- Can be scheduled/active only with `starts_on`.
- Future schedule is calculated from its weekly plan items.
- Current and past occurrences are represented by `scheduled_workouts`.
- Edits affect future virtual schedule only.
- Edits do not rewrite materialized historical snapshots.

### Training Plan Item

A weekly schedule row. It says which workout belongs to which weekday and role.

Rules:

- Belongs to one training plan.
- References one workout in the same plan and business.
- Uses weekday strings.
- Can represent more than primary/alternative.
- Ordering is explicit through `position`.

### Scheduled Workout

A materialized workout occurrence for a client on a concrete date.

Rules:

- Generated by system workflows, not normal CRUD.
- Exists for current and past dates that need history, adherence, or logging.
- Stores a `planned_snapshot`.
- Is the stable bridge between a virtual weekly plan and actual logged sessions.
- Is not created for every future date.

### Workout Session

The actual client execution/logging record.

Rules:

- Usually belongs to a `scheduled_workout`.
- Preserves a planned snapshot.
- Contains performed exercises and performed sets.
- Completed sessions are historical facts and must not be rewritten by plan edits.

## Schema Design

### `training_plans`

Keep one table for templates and client plans. Make the concept explicit.

Fields:

```text

```

Enums:

```text
kind:
  template
  client_plan

status:
  draft
  active
  archived
```

Field rules:

- `business_id` is required internally and never returned.
- `author_id` is required internally.
- `kind = template` requires `client_id IS NULL`.
- `kind = template` requires `starts_on IS NULL`.
- `kind = template` requires `ends_on IS NULL`.
- `kind = client_plan` requires `client_id IS NOT NULL`.
- `kind = client_plan, status = draft` allows `starts_on IS NULL`.
- `kind = client_plan, status = active` requires `starts_on IS NOT NULL`.
- `ends_on` is optional for client plans. If present, `ends_on >= starts_on`.
- `source_template_id` is optional and coach-facing only.
- Delete `original_template_id` and replace it with `source_template_id`.
- Delete `rest_days`.
- Delete automatic copy-name generation.

Indexes and constraints:

```text
index business_id
index business_id, kind
index business_id, client_id
index business_id, status
index business_id, source_template_id
check kind in ('template', 'client_plan')
check status in ('draft', 'active', 'archived')
check template_has_no_client
check template_has_no_dates
check client_plan_has_client
check active_client_plan_has_start
check valid_date_range
```

Only one active current client plan should be allowed per client unless a future product decision explicitly supports multiple concurrent training plans.

Use a partial unique index for active client plans:

```text
unique business_id, client_id
where kind = 'client_plan' and status = 'active' and archived_at is null
```

- Allow multiple alternatives/accessories/conditioning/mobility/rehab rows per day.


Validation:

- `id` is the stable public planned set identifier.
- `set_index` is required and non-negative.
- `set_type` is required.

- Numeric values must be non-negative unless a future assisted-load use case explicitly allows negative load.

- If `target_rpe_min` and `target_rpe_max` are both present, min must be less than or equal to max.
- RPE values must be between 1 and 10.
- RIR values must be non-negative.
- `load_value` requires a meaningful `load_unit`.
- `distance_value` requires a meaningful `distance_unit`.
- `duration_seconds` must be positive when used as the only target.
- Planned set rows are copied when a template is duplicated or prepared for a client.

The principle is:

```text
Validate data shape strictly. Validate coaching intent lightly.
```

### `scheduled_workouts`

Materialized current/past workout occurrence.

Fields:

```text
id
business_id
client_id
training_plan_id
training_plan_item_id
workout_id
scheduled_date
day
role
position
status
planned_snapshot
materialized_at
completed_at
missed_at
skipped_at
skipped_reason
rescheduled_from_date
rescheduled_to_date
status_changed_by_id
inserted_at
updated_at
```

Enums:

```text
status:
  scheduled
  in_progress
  completed
  skipped
  missed
  canceled
```

Rules:

- These rows are generated by system workflows.
- No normal `POST /scheduled_workouts` endpoint.
- Future workout calendar views are virtual and should not create rows.
- Today's occurrences are materialized before the client can log against them.
- If the daily materializer has not run, first access must lazily materialize the needed row.
- Day-end job marks unlogged materialized occurrences as missed after the business-local day closes.
- Plan edits do not mutate existing materialized `planned_snapshot`.
- `planned_snapshot` stores enough data to render and compare planned vs actual even after the plan changes.
- Unique per client/date/training plan item:

```text
unique business_id, client_id, scheduled_date, training_plan_item_id
```

Materialization policy:

```text
future = virtual from current active weekly plan
today = materialized at start of day or first access
past = materialized historical rows
```

The day-end job is a closer, not the only creator. This avoids capturing a late edited plan as the historical prescription for a workout the client saw earlier.

- Starting a session sets the scheduled workout status to `in_progress`.
- Completing a session sets scheduled workout status to `completed`.
- Discarding a session does not delete historical scheduled workout rows.
- `planned_snapshot` is immutable after session completion except by explicit admin correction.

- RIR values must be non-negative.

## Core Workflows

### Create Template

Endpoint:

```text
POST /v1/coach/training_plan_templates
```

Creates `training_plans.kind = template`.

Request accepts:

```text
name
description
goal
level
status
```

Request rejects:

```text
business_id
author_id
client_id
source_template_id
starts_on
ends_on
rest_days
```

### Duplicate Template

Endpoint:

```text
POST /v1/coach/training_plan_templates/:id/duplicate
```

Requires:

```text
name
```

Rules:

- No generated copy names.
- Deep-copy workouts, workout elements, planned set rows, and training plan items.
- Preserve schedule semantics, but assign new row IDs and planned set IDs.

### Prepare Client Plan

Endpoint:

```text
POST /v1/coach/training_plan_templates/:id/prepare-for-client
```

Request:

```text
client_id
name
```

Rules:

- Creates `kind = client_plan`.
- Creates `status = draft`.
- Sets `client_id`.
- Does not require `starts_on`.
- Does not create `scheduled_workouts`.
- Hidden from client unless product later decides to show draft plans.

### Activate Client Plan

Endpoint:

```text
POST /v1/coach/client_training_plans/:id/activate
```

Request:

```text
starts_on
```

Rules:

- Requires client plan draft.
- Sets `status = active`.
- Sets `starts_on`.
- Archives any currently active client plan for the same client, or returns a conflict if product chooses manual archive. This spec chooses archive previous active plan automatically because the target product supports one active weekly plan per client.
- Does not bulk-create future scheduled workouts.
- May materialize today's workout occurrences if `starts_on <= today`.

### Future Calendar View

Endpoint:

```text
GET /v1/client/workout_schedule
GET /v1/coach/clients/:client_id/workout_schedule
```

Rules:

- Combines materialized historical rows with virtual future rows.
- Past and today read from `scheduled_workouts`.
- Future dates calculate from the active weekly plan and its `training_plan_items`.
- Virtual rows should include a stable deterministic key but no database ID.

### Daily Materialization

Job:

```text
Easy.Training.MaterializeDailyWorkouts
```

Rules:

- Runs early each day for active client plans.
- Creates scheduled workout rows for that date from the active weekly plan.
- Is idempotent.
- Can be called lazily on first schedule/session access for a client/date.
- Stores `planned_snapshot`.

### Day-End Closing

Job:

```text
Easy.Training.CloseDailyWorkouts
```

Rules:

- Finds materialized scheduled workouts for the date that are still `scheduled`.
- Marks them `missed` after the configured local-day cutoff.
- Does not create snapshots as its primary job.
- Does not touch completed, skipped, canceled, or in-progress rows.

### Start Workout Session

Endpoint:

```text
POST /v1/client/scheduled_workouts/:id/start
```

Rules:

- Requires the scheduled workout to belong to the authenticated client.
- Creates a workout session with the scheduled workout snapshot.
- Creates performed exercises and performed sets eagerly from the snapshot.
- Sets scheduled workout status to `in_progress`.

### Complete Workout Session

Endpoint:

```text
POST /v1/client/workout_sessions/:id/complete
```

Rules:

- Validates performed exercises/sets.
- Sets session state to `completed`.
- Sets scheduled workout status to `completed`.
- Computes `completed_as_prescribed`.
- Stores notes, pain, RPE/RIR, and duration.

### Skip Workout

Endpoint:

```text
POST /v1/client/scheduled_workouts/:id/skip
```

Rules:

- Sets scheduled workout status to `skipped`.
- Stores reason.
- Does not require a workout session.

### Reschedule Workout

Endpoint:

```text
POST /v1/coach/scheduled_workouts/:id/reschedule
```

Rules:

- Only works on materialized future/today rows.
- Preserves `rescheduled_from_date`.
- Updates `scheduled_date`.
- Does not affect completed sessions.

- Generated OpenAPI and controller behavior must agree.

## Response Boundary

Never expose:

```text
business_id
internal audit actor IDs unless represented as public coach/client summaries
raw database tenant internals
```

Coach responses may expose:

```text
id
client summary
source template summary
author summary
status
starts_on
ends_on
training plan items
scheduled workout statuses

```

Client responses expose:

```text

own schedule
own scheduled workouts
own sessions

```

Client responses should hide:

```text

author_id
creator_id
client_id when the authenticated client is implied
```

## Database Migration Strategy

Because this is not production, prefer direct migrations and schema rewrites over compatibility layers.

Steps:

1. Add `kind`, `source_template_id`, `starts_on`, `ends_on`, `goal`, `level`, lifecycle fields to `training_plans`.
2. Backfill existing rows:
   - `client_id IS NULL` becomes `kind = template`.
   - `client_id IS NOT NULL` becomes `kind = client_plan`.
   - `original_template_id` becomes `source_template_id`.
3. Remove `original_template_id`.
4. Remove `rest_days`.
5. Rename `training_plan_items.workout_type` to `role`.
6. Add `training_plan_items.position`, `label`, `notes`, and `is_optional`.
7. Add workout metadata fields.
8. Add workout element metadata fields and `training_plan_id`.
9. Create `planned_sets` and migrate embedded planned set data into rows.
10. Create `scheduled_workouts`.
11. Create `scheduled_workout_events`.
12. Add schedule/session links to `workout_sessions`.
13. Create `performed_exercises`.
14. Refactor `performed_sets` to belong to performed exercises and include planned set ID, planned snapshot UID, and status fields.
15. Replace old controllers/routes/OpenAPI schemas directly.
16. Replace old tests directly.

## Test Strategy

Prioritize API boundary tests over context-only tests.

### Template API Tests

- Create template succeeds with allowed fields.
- Create template rejects `business_id`, `client_id`, dates, and unknown fields.
- Duplicate template requires user-provided name.
- Duplicate creates independent workouts, elements, planned set rows, and plan items.
- Template responses never include `business_id`.
- Template list is tenant-scoped.

### Client Plan API Tests

- Prepare client plan from template without `starts_on`.
- Prepared draft is not visible to client.
- Activate client plan requires `starts_on`.
- Activating one plan archives the prior active client plan.
- Client plan edits do not mutate source template.
- Cross-tenant client/template IDs return safe not found.

### Weekly Schedule Tests

- Multiple workouts can exist on one weekday.
- Only one primary workout can exist per weekday.
- Same-day ordering follows `position`.
- Missing weekday means rest.
- Invalid day/role is rejected at OpenAPI and changeset boundaries.

### Scheduled Workout Tests

- Future schedule is virtual and does not create rows.
- Today's schedule materializes rows with planned snapshots.
- Materialization is idempotent.
- Plan edits do not rewrite existing materialized snapshots.
- Day-end closer marks unlogged scheduled rows as missed.
- Skip stores status and reason.
- Reschedule preserves original date.

### Workout Logging Tests

- Starting a scheduled workout creates a session.
- Starting creates performed exercises and performed sets from snapshot.
- Completing as prescribed sets `completed_as_prescribed`.
- Modified reps/load/duration/distance are saved.
- Skipped sets are saved.
- Added sets are saved.
- Exercise substitution stores prescribed and actual exercises.
- Changed exercise order is saved.
- RPE, RIR, pain, soreness, duration, and notes are saved.
- Completed session preserves planned snapshot after plan edits.

### Tenant Isolation Tests

- Coach cannot access another business template.
- Coach cannot prepare plan for another business client.
- Client cannot view another client's scheduled workout.
- Client cannot start another client's session.
- Performed exercise/set cannot attach to another business session.
- Exercise from another business cannot be prescribed unless global exercise visibility permits it.

### OpenAPI Tests

- Generated OpenAPI includes every new endpoint.
- Request schemas reject unknown fields.
- Response schemas omit `business_id`.
- Coach and client schemas differ where required.
- Swagger UI renders against generated OpenAPI.

## Refactor Completion Criteria

The refactor is complete only when:

- Old training plan routes are removed or replaced.
- Old `rest_days` behavior is gone.
- Templates and client plans are explicit through `kind`.
- Client plan drafts do not require dates.
- Active client plans require `starts_on`.
- Future schedule is virtual.
- Current/past schedule history is materialized.
- Workout sessions link to scheduled workouts and preserve planned snapshots.
- Performed exercises and performed sets represent substitutions, skipped sets, added sets, and planned-vs-actual data.
- No training API response includes `business_id`.
- OpenAPI specs exist and validate request bodies for changed endpoints.
- API boundary tests cover success, validation failure, auth failure, and tenant isolation.
